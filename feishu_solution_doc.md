# 飞书话题群智能服务协同助手

> **一句话概括**
>
> 飞书话题群智能服务协同助手，面向企业内部高频咨询和跨团队协同场景，提供“AI 秒级解答 + 一键转人工 + 专属服务群协同 + 自动归档回收”的完整闭环。

---

## 安装方案

> 本方案基于飞书开放平台机器人、Aily 能力、多维表格与群聊协同能力构建，当前项目以 Python 服务形式运行，通过 WebSocket 长连接监听飞书事件，无需额外暴露公网回调地址。

| 部署模块 | 说明 |
| --- | --- |
| 运行环境 | 在一台可稳定联网的 Windows 或 Linux 主机上运行机器人进程 |
| 配置方式 | 通过 `.env` 配置飞书应用凭证、可触发群、部门负责人、多维表格等参数 |
| 使用入口 | 使用飞书话题群作为统一入口，用户直接在群里新建话题提问 |
| 数据沉淀 | 通过多维表格承接统计与归档结果，形成后续分析与复盘基础 |

这套方案适合先在小范围业务群试点，再逐步扩展到 IT 支持、产品咨询、财务对账、商务协同等内部服务场景。

---

## 方案简介

用户无需跳转外部工单系统，也无需先理解复杂流程，只需要在授权群内发起一个话题，系统就会立即回复“思考中”占位卡片，并在完成 AI 分析后把答案原地更新到同一条消息中。若 AI 回答未能解决问题，用户可直接点击部门按钮，系统自动创建专属服务群、拉入提问人和对应负责人、转发原始问题，并在问题结束后执行归档与解散。

相较于传统“先填单、再分派、再找人、再补上下文”的流程，这个项目把问题入口、AI 问答、人工介入、归档统计和群生命周期管理全部收敛在飞书内完成。

| 核心闭环 | 对应动作 |
| --- | --- |
| 问题进入 | 用户在授权群内发起话题 |
| AI 响应 | 系统先返回占位卡片，再更新真实回答 |
| 人工升级 | 点击部门按钮后自动建群、拉人、转发上下文 |
| 结果沉淀 | 自动归档、多维表格记录、群自动回收 |

## 适用场景

### 1. 内部通用支持类场景

- IT 故障咨询
- 产品功能使用答疑
- 商务合作沟通分流
- 财务对账与报销问题协同

这类场景通常具有两个特点：高频、重复问题多；少量复杂问题又需要快速拉齐上下文和责任人。本项目正好覆盖这两段流程。

### 2. 需要图片或文件辅助判断的咨询场景

项目已支持从飞书消息中提取文本、图片与文件资源，并将附件上传到 Aily 侧参与回答，适合截图报错、制度文档解读、表格或附件辅助判断等场景。

### 3. 需要轻量工单化但不想引入重系统的团队

如果团队当前还不想上完整服务台，但已经明显感受到“群消息淹没问题、重复问答耗时、拉群靠人工、结束后无人收尾”的痛点，这套方案可以作为轻量级服务协同中台。

## 实现概览

| 步骤 | 系统动作 |
| --- | --- |
| 1 | 用户在指定群内发送话题首条消息 |
| 2 | 机器人过滤非群消息、机器人自身消息和话题回复，仅处理真正的新问题 |
| 3 | 系统立即发送“AI 思考中”卡片，避免用户等待无反馈 |
| 4 | 后端解析文本、图片、文件等输入，并调用 Aily 生成回答 |
| 5 | 机器人将真实回答更新回原卡片，同时附带“AI 已解决”与“联系人工客服”按钮 |
| 6 | 若用户点击某个部门按钮，系统自动创建服务群，拉入提问者、点击者和对应负责人 |
| 7 | 系统向新群发送欢迎卡片，并转发原始提问消息，保证上下文无损衔接 |
| 8 | 服务群进入自动巡检状态，闲置后提醒确认解散，归档聊天记录并自动删群 |
| 9 | 关键过程同步记录到多维表格，用于统计 AI 拦截率、人工介入情况和归档结果 |

## 服务闭环流程

> **阶段一：智能接待**
>
> 用户在话题群发问后，机器人会优先完成快速响应，再异步执行 AI 分析。这种“先占位、后更新”的机制能够显著改善用户体感，避免看起来像机器人没有收到请求。

> **阶段二：AI 自主解决**
>
> 如果 Aily 返回的答案足够解决问题，用户可以直接点击“AI 回答已解决我的问题”。系统会把该记录标记为 AI 自动解决，并更新统计表中的解决方式字段。

> **阶段三：一键转人工**
>
> 如果问题超出 AI 当前能力，用户无需重新描述问题，只需要点击对应部门按钮。系统会：
>
> - 根据卡片消息 ID 和部门键做幂等控制
> - 避免多人同时点击造成重复建群
> - 自动创建服务群并去重拉人
> - 推送欢迎卡片，说明当前服务团队和使用方式
> - 转发原问题消息，把上下文完整带入新群

> **阶段四：归档与释放**
>
> 服务群创建后会被自动纳入追踪。若长时间无人发言，机器人会发送“是否确认解散”卡片。确认后系统会整理群聊历史、写入多维表格，并在倒计时结束后自动解散群聊；若 24 小时无人响应，也会触发兜底归档和回收。

## 方案亮点

| 维度 | 内容 |
| --- | --- |
| 对用户足够轻 | 用户只需要会在群里发消息，不需要额外学习工单系统。复杂流程由机器人在后台接管。 |
| 对处理人足够准 | 人工介入并不是“把人扔进大群里找问题”，而是自动生成一个带着原始问题上下文的专属服务群，减少重复沟通。 |
| 对系统运行足够稳 | 项目在多个关键节点实现了防重和容错。 |
| 对管理层可统计 | 当前项目已将关键过程沉淀到多维表格中。 |

### 运行稳定性细节

- 消息处理去重，避免飞书重放事件导致重复回复
- 卡片点击防抖，避免短时间重复点击
- 建群缓存持久化，避免同一卡片同一部门重复建群
- 服务群状态持久化，进程重启后仍能恢复自动解散状态
- `.env` 变更自动热重启，降低配置调整成本

### 当前已沉淀的统计字段

| 统计字段 |
| --- |
| 提问内容 |
| 提问时间 |
| 所在群 ID |
| AI 回答内容 |
| 解决方式 |
| 转入部门 |
| 操作人 |
| 服务群归档记录与消息条数 |

这意味着团队可以继续向“响应效率、AI 拦截率、人工接入占比、归档质量”几个方向扩展数据看板。

## 典型角色视角

| 角色 | 体验与价值 |
| --- | --- |
| 一线用户 | 用户在群里发出问题后，首先看到的是机器人立刻返回的占位卡片，而不是长时间静默等待。几秒内，卡片被更新为 AI 回答内容。如果答案有效，用户一键确认即可结束。如果答案不够，直接点“技术支持”或“产品咨询”等按钮，系统自动帮用户建群、找人、带上下文，用户不需要再重新描述一遍问题。 |
| 处理人 | 负责人被自动拉入服务群时，群里已经包含提问人、发起人工协同的用户、对应部门负责人、原始问题消息、机器人欢迎卡片与使用说明。处理人收到的不是碎片化咨询，而是一个已经完成路由、上下文完整、可持续跟进的小闭环协同空间。 |
| 管理者 | 管理者最直接获得三类价值：用 AI 拦截高频重复问题，降低一线支持压力；用自动拉群和自动归档减少协同过程中的人工操作；用多维表格沉淀过程数据，为后续做数据看板和服务优化提供基础。 |

从项目当前实现看，它已经具备成为“轻量级内部服务台入口”的基础能力，后续只需继续补充知识库、部门路由策略和数据分析口径，就能进一步演进为更完整的服务运营平台。

## 总结

> 这不是一个单点问答机器人，而是一套围绕“内部问题受理”搭建的飞书内闭环协同方案。它把 AI 问答、人工升级、专属群协同、归档统计和群回收串成一条可运行的业务链路，适合企业内部服务团队快速上线并持续演进。
